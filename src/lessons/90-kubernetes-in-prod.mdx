<section class="title-center-body-left">
## <Icon name="i-tabler-alert-triangle" />Failure in Production

<p class="block-title">In real systems:</p>

- Pods crash
- Images fail to pull
- Nodes run out of memory
- Deployments misconfigure secrets

Good engineers expect failure.
They design for it.
</section>

<section>
## <Icon name="i-tabler-arrows-exchange" />Controlled Change: Rolling Updates

When you change an image:

```bash
kubectl set image deployment/quote-app app=quote-app:v2
```

Kubernetes does not destroy everything.

It rolls forward gradually.
</section>

<section>
## <Icon name="i-tabler-list-details" />Observe a Rollout

```bash
kubectl rollout status deployment quote-app
kubectl rollout history deployment quote-app
```

You can:

- Monitor progress
- Inspect revision history
</section>

<section>
## <Icon name="i-tabler-rotate-2" />Rollback

If something breaks:

```bash
kubectl rollout undo deployment quote-app
```

<p class="block-title">Production mindset:</p>

_Change must be reversible._
</section>

<section>
## <Icon name="i-tabler-brain" />Think Like an Engineer

<p class="block-title">Ask yourself:</p>

- What is my blast radius?
- What layer isolates failure?
- What is my rollback plan?
- What happens if the node dies?
- Where is persistence guaranteed?

_This is what differentiates operators from YAML writers._
</section>
